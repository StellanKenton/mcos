# DrvLayer 新驱动目录构建规范

当需要在 `drvlayer` 下新增一个新的驱动目录时，使用本文档约束目录结构、命名方式、职责边界和接入方式。本文档参考 `DrvGpio` 的组织方式，目标是让后续新增的 `DrvXxx` 模块都保持一致的抽象层次，避免应用层直接依赖 MCU SDK 或板级细节。

## 目标

在新增驱动目录时，应优先建立稳定的公共驱动接口，再把 MCU、芯片、开发板和外设资源差异封装到 BSP 层或 port 层。应用层只依赖 `drvXxx*` 接口，不直接调用 MCU SDK、寄存器定义或板级初始化细节。

新增驱动模块后，调用关系应尽量保持为：

- 应用层 -> `drvXxx*`
- `DrvXxx` 公共层 -> port 层或 BSP 层
- BSP 层 -> MCU SDK / 寄存器 / 板级资源

## 适用范围

本文档适用于 `USER/Rep/drvlayer` 下所有后续新增的驱动目录，例如：

- `DrvSpi`
- `DrvI2c`
- `DrvPwm`
- `DrvAdc`
- `DrvFlash`
- `DrvCan`

如果某个驱动模块因为硬件差异较大需要额外文件，也必须先满足本文档中的公共层和边界规则，再在局部扩展。

## 目录与文件建议

新增驱动目录时，优先采用下面这组基础文件：

- `drvxxx.h`：公共驱动头文件，对应用层暴露稳定接口、状态码、必要类型定义。
- `drvxxx.c`：公共驱动实现文件，负责参数校验、状态维护、公共逻辑和对 BSP/port 层转发。
- `drvxxx_port.h` 或 `drvxxx_portmap.h`：项目级逻辑资源映射、逻辑端口枚举、逻辑通道定义或模块级开关配置。
- `drvxxx_port.c`：port 层绑定文件，负责把公共层和 BSP 实现连接起来。
- `xxx.md`：该驱动模块的接入说明、提示词模板、架构说明或资源映射说明，主要用于生成bsp和drv层之间连接的接口。

推荐目录示例：

```text
DrvXxx/
    drvxxx.h
    drvxxx.c
    drvxxx_port.h
    drvxxx_port.c
    drvxxx_prompt.md      // Optional integration prompt or notes
Bsp/
    bspxxx.h
    bspxxx.c
```

## 命名规范

- 驱动目录名统一使用 `drvXxx` 风格。
- 公共接口函数统一使用 `drvXxx*` 前缀。
- BSP 接口函数统一使用 `bspXxx*` 前缀。
- 类型名遵循仓库规则，结构体类型使用 `st` 前缀，枚举类型使用 `e` 前缀。
- 模块内全局变量和静态文件作用域变量使用 `g` 前缀。
- 临时局部变量使用 `l` 前缀。
- 宏名称仅在确实需要宏时使用全大写加下划线。

例如：

- 目录：`DrvSpi`
- 公共头文件：`drvspi.h`
- 公共实现：`drvspi.c`
- 逻辑映射：`drvspi_port.h`
- 绑定文件：`drvspi_port.c`
- BSP 文件：`bspspi.h`、`bspspi.c`

## 分层职责

### 1. 公共层 `drvxxx.h` 和 `drvxxx.c`

公共层只负责下面这些内容：

- 定义稳定的对外 API。
- 定义模块统一的状态码、错误码或结果语义。
- 进行模块边界参数校验。
- 维护与硬件无关的公共状态。
- 在需要时维护通用缓冲、状态机、缓存或软件回退逻辑。
- 调用 BSP/port 层钩子，不直接访问 MCU 寄存器或 SDK。

公共层不应承担下面这些内容：

- GPIO、UART、SPI、DMA、IRQ、时钟等 MCU 专有配置细节。
- 板级具体引脚号、通道号、外设实例号。
- 应用层业务流程或上层策略。

### 2. 逻辑映射层 `drvxxx_port.h` 或 `drvxxx_portmap.h`

该层负责项目级逻辑资源定义，例如：

- 逻辑端口枚举
- 逻辑通道编号
- 逻辑设备索引
- 模块级开关配置
- 与具体板级资源对应的抽象编号

如果应用层需要通过“逻辑资源 ID”访问底层硬件，这一层通常是必须的。

### 3. 绑定层 `drvxxx_port.c`

该层负责：

- 建立公共层和 BSP 层之间的钩子表。
- 统一管理 BSP 回调函数绑定。
- 尽量避免在公共层直接写死 BSP 依赖。

如果模块非常简单，并且当前仓库同类驱动已经把绑定写在 `drvxxx.c` 中，可以沿用现有局部风格；但默认推荐保留独立的 port 绑定层。

### 4. BSP 层 `bspxxx.h` 和 `bspxxx.c`

BSP 层只负责硬件相关实现，例如：

- 时钟使能
- 端口和引脚配置
- 外设实例选择
- 中断号、DMA 通道、优先级
- 轮询、IT、DMA 等具体收发方式
- 寄存器访问或 SDK 调用
- 与特定 MCU 或板卡绑定的资源映射

所有 MCU 和板级差异必须封装在 BSP 层内部，不能泄漏到公共层和应用层。

## 新驱动目录的最低要求

新增 `DrvXxx` 目录时，至少应满足以下要求：

1. 应用层只调用 `drvXxx*` 接口。
2. 模块对外暴露的 API 名称、参数和返回值语义清晰稳定。
3. 公共层不直接包含 MCU 专有寄存器或 SDK 细节。
4. 如果存在逻辑资源枚举，应集中放在 `drvxxx_port.h` 或 `drvxxx_portmap.h`。
5. 如果存在 BSP 钩子表，应明确哪些钩子是必需的，哪些是可选的。
6. 参数校验放在公共层边界执行，避免无声失败。
7. 失败路径要返回明确状态，不能静默忽略硬件或资源错误。
8. 新增文件必须符合仓库的编码规则、缩进和注释约束。
9. 模块必须附带一份简短的说明文档或提示词文档，说明结构、接入方式和验证要点。

## 接口设计规则

在设计新的 `drvXxx` 接口时，优先遵守以下规则：

- 先定义最小可用接口，再决定是否增加异步、DMA、缓存或扩展能力。
- 如果某些能力并非所有平台都支持，接口仍可保留，但必须通过明确状态码表达 `unsupported`。
- 公共层接口应尽量稳定，不要让应用层感知底层外设差异。
- 当模块需要多个逻辑实例时，优先使用逻辑枚举或逻辑句柄，而不是直接暴露物理端口编号。
- 如果输入参数包含缓冲区和长度，必须在模块边界做空指针和长度检查。
- 如果模块维护内部状态，应显式定义初始化前、初始化后、忙状态和错误状态的行为。

## BSP 钩子表建议

如果驱动需要通过钩子表访问 BSP，实现方式建议参考 `DrvGpio` 或 `DrvUart` 的风格，定义一个模块专属接口结构体，例如：

```c
typedef struct stDrvXxxBspInterface {
    drvXxxBspInitFunc init;
    drvXxxBspReadFunc read;
    drvXxxBspWriteFunc write;
    drvXxxBspControlFunc control;
} stDrvXxxBspInterface;
```

设计要求：

- 必需钩子必须在初始化时统一检查完整性。
- 可选钩子在公共层调用前判断是否为 `NULL`。
- 如果可选钩子缺失，应返回清晰状态码或采用已定义的软件回退逻辑。
- 钩子表成员名称应与公共接口语义一致，不要混入板级实现细节命名。

## 逻辑资源定义规则

如果模块需要逻辑资源枚举，建议按下面方式组织：

- 逻辑资源名称从项目视角命名，而不是从芯片视角命名。
- 逻辑资源枚举定义应独立于公共接口声明。
- 如果未来可能扩展，保留 `MAX` 结尾项。
- 如果模块存在无效值或错误值语义，应显式定义。

例如：

```c
typedef enum eDrvSpiPortMap {
    DRVSPI_SENSOR = 0,
    DRVSPI_FLASH,
    DRVSPI_MAX,
} eDrvSpiPortMap;
```

这里的 `DRVSPI_SENSOR` 和 `DRVSPI_FLASH` 表达的是项目逻辑资源，而不是底层具体使用 `SPI0` 还是 `SPI2`。

## 文档要求

每个新的 `DrvXxx` 目录，建议至少附带一份 Markdown 文档，用于说明：

- 模块目标
- 公共层职责
- BSP 层职责
- 逻辑资源定义方式
- 需要实现的 BSP 接口
- 绑定方式
- 生成代码时的输入输出约束
- 验证方法和剩余硬件风险

如果该模块未来会频繁通过 AI 生成或补全 BSP 代码，推荐直接提供一份 `drvxxx_prompt.md` 或 `bspxxx.md`，明确提示词模板，减少后续重复约束。

## 推荐工作流程

新增 `DrvXxx` 目录时，建议按下面顺序执行：

1. 先读取 `rule.md`，确认当前仓库的编码、命名和边界规则。
2. 检查 `drvlayer` 下现有相似模块，优先复用局部主流结构。
3. 确认该模块是否需要逻辑资源枚举或逻辑端口映射。
4. 先定义 `drvxxx.h` 中的稳定公共接口和状态码。
5. 再实现 `drvxxx.c` 中的参数校验、状态管理和转发逻辑。
6. 需要逻辑映射时，补充 `drvxxx_port.h` 或 `drvxxx_portmap.h`。
7. 需要独立绑定层时，补充 `drvxxx_port.c`。
8. 需要新增 BSP 文件时，再实现 `bspxxx.h` 和 `bspxxx.c`。
9. 最后补充 Markdown 说明文档，写清楚资源映射、约束和验证方式。

## 新驱动目录提示词模板

下面这份模板可直接用于后续让 AI 生成新的 `drvXxx` 模块：

```md
你正在 Windows 环境下。

任务要求：
1. 先检查 `rule.md`，并遵守仓库中的编码规范。
2. 检查 `drvlayer` 下与目标模块最接近的现有目录，并复用局部主流结构。
3. 在 `drvlayer` 下创建新的 `drvXxx` 目录，至少包含公共层文件。
4. 公共层对应用层暴露稳定的 `drvXxx*` 接口，应用层不能直接调用 MCU SDK。
5. 所有 MCU、板级、时钟、引脚、DMA、中断和寄存器细节都必须封装在 BSP 层内部。
6. 如果该模块需要逻辑资源编号，先创建 `drvxxx_port.h` 或 `drvxxx_portmap.h`，再让 BSP 与其对应。
7. 如果模块通过钩子表连接 BSP，实现时要明确必需钩子和可选钩子，并在初始化或调用前做校验。
8. 参数校验、公共状态维护和错误码归一化应放在 `drvxxx.c` 中。
9. 改动尽量小，不要重构无关模块。
10. console相关的内容要使用宏定义开关，并且默认关闭。
11. console相关的注册及对应的函数要放在port中，尽量高内聚低耦合，尽量合并同类程序，减少重复代码。

需要产出：
1. 新驱动目录下的公共头文件和实现文件。
2. 逻辑映射文件或绑定文件（如果该模块需要）。
3. BSP 文件或对现有 BSP 文件的补充（如果该模块需要）。
4. 一份 Markdown 说明文档，描述目录结构、接口语义、逻辑资源和硬件资源的映射关系。
5. 一段验证说明，描述已经检查了什么，以及还剩哪些硬件相关风险。

实现约束：
- 代码中的注释必须使用英文。
- 每个文件只负责一个清晰职责。
- 不要增加占位接口、伪实现或与任务无关的抽象层。
- 保持现有 `drvXxx*` 调用方式和仓库整体风格一致。
- 公共层不允许泄漏 MCU 专有实现细节。
- 需要失败语义时，必须返回明确状态，而不是静默忽略。
```

## 期望结果

构建完成后的新驱动目录，应满足以下特征：

- 目录职责清晰，应用层、公共层、绑定层和 BSP 层边界明确。
- 命名统一，后续模块可以横向类比和复用。
- 逻辑资源和物理资源解耦，不把芯片细节传播到上层。
- 新 MCU 或新板卡接入时，主要改动集中在 BSP 或 port 层。
- 公共接口稳定，上层业务不需要因为硬件切换而重写。
