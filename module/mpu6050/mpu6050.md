# MPU6050 Module

## Overview

- This module provides a blocking MPU6050 driver built on top of the shared IIC drv abstraction.
- Each `stMpu6050Device` instance owns its own runtime IIC binding, so different devices can safely select different IIC implementations.
- The port layer dispatches each device to either the hardware IIC drv or the software IIC drv according to that binding.
- The default binding is hardware IIC on `DRVIIC_BUS0`. Use the port helpers to switch a specific device to software IIC when needed.
- The default device address is `0x68`. Use `0x69` when the MPU6050 `AD0` pin is pulled high.

## Files

- `mpu6050.h`: Public API, device structure, status codes, and raw sample type.
- `mpu6050.c`: Device initialization, register access helpers, sleep control, and raw sample reads.
- `mpu6050_port.h`: Project-level runtime IIC binding helpers and delay settings.
- `mpu6050_port.c`: Binding layer that routes each device to the selected hardware or software IIC drv adapter.

## Initialization Flow

1. Call `mpu6050GetDefaultConfig()`.
2. Adjust `address`, `sampleRateDivider`, `dlpfConfig`, `accelRange`, or `gyroRange` if needed.
3. Keep the default hardware binding or call `mpu6050PortSetSoftwareIic()` or `mpu6050PortSetHardwareIic()` for the selected device.
4. Call `mpu6050Init(&device)` once before normal read or write operations.

## Public API

- `mpu6050Init(&device)`: Initializes the selected device bus, verifies `WHO_AM_I`, resets the device, and applies the selected configuration.
- `mpu6050ReadWhoAmI(&device, ...)`: Probes the selected device on its configured bus.
- `mpu6050ReadRawSample(&device, ...)`: Reads one 14-byte sensor frame containing accel, temperature, and gyro raw values.
- `mpu6050ReadTemperatureCentiDegC(&device, ...)`: Converts the raw temperature sample to centi-degrees Celsius.
- `mpu6050SetSleepEnabled(&device, ...)`: Enables or disables the sleep bit in `PWR_MGMT_1`.
- `mpu6050ReadRegister(&device, ...)` and `mpu6050WriteRegister(&device, ...)`: Expose direct single-register access for the selected device.

## Verification Notes

- Build verification confirms that the module and port binding compile into the current firmware.
- Hardware verification is still required to confirm pull-up sizing, signal integrity, and the expected `WHO_AM_I` value on the target board.
